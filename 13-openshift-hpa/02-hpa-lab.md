# OpenShift – HPA Lab Setup

## What We Are Doing

Deploy a Node.js web application (`i27devopsb8/node:livereadv1`) with resource requests set, then create an HPA that automatically scales replicas based on CPU utilization. We use a web app so load can be generated by hitting an HTTP endpoint in a loop.

---

## Prerequisites

```bash
oc adm new-project i27-hpa-demo
oc project i27-hpa-demo
```

Apply SCC (required for this image):

```bash
oc adm policy add-scc-to-user anyuid -z default -n i27-hpa-demo
```

---

## Step 1 — Deploy the Application

Resource requests are **mandatory** for HPA to work. Without them, HPA cannot calculate CPU utilization percentage.

**`hpa-deployment.yaml`**

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: hpa-demo
  namespace: i27-hpa-demo
spec:
  replicas: 1          # start with 1 — HPA will scale from here
  selector:
    matchLabels:
      app: hpa-demo
  template:
    metadata:
      labels:
        app: hpa-demo
    spec:
      containers:
      - name: nodeapp
        image: i27devopsb8/node:livereadv1
        ports:
        - containerPort: 3000
        resources:
          requests:
            cpu: "100m"      # HPA calculates utilization against this value
            memory: "128Mi"
          limits:
            cpu: "200m"
            memory: "256Mi"
---
apiVersion: v1
kind: Service
metadata:
  name: hpa-demo-svc
  namespace: i27-hpa-demo
spec:
  selector:
    app: hpa-demo
  ports:
  - name: http
    protocol: TCP
    port: 80
    targetPort: 3000
  type: ClusterIP
---
apiVersion: route.openshift.io/v1
kind: Route
metadata:
  name: hpa-demo-route
  namespace: i27-hpa-demo
spec:
  to:
    kind: Service
    name: hpa-demo-svc
  port:
    targetPort: http
```

Apply:

```bash
oc apply -f hpa-deployment.yaml
```

Verify pod is running:

```bash
oc get pods -n i27-hpa-demo
```

Store the route URL for later:

```bash
ROUTE_URL=http://$(oc get route hpa-demo-route -o jsonpath='{.spec.host}' -n i27-hpa-demo)
echo $ROUTE_URL
```

Test the app responds:

```bash
curl $ROUTE_URL
```

## Step 2 — Create HPA

### Option A — Using CLI (`oc autoscale`)

```bash
oc autoscale deployment hpa-demo \
  --min=1 \
  --max=5 \
  --cpu-percent=50 \
  -n i27-hpa-demo
```

| Flag | Value | Meaning |
|------|-------|---------|
| `--min` | `1` | Never scale below 1 replica |
| `--max` | `5` | Never scale above 5 replicas |
| `--cpu-percent` | `50` | Scale up when average CPU exceeds 50% of request |

---

### Option B — Using YAML

**`hpa.yaml`**

```yaml
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: hpa-demo
  namespace: i27-hpa-demo
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: hpa-demo          # must match the Deployment name exactly
  minReplicas: 1
  maxReplicas: 5
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 50    # target 50% of CPU request across all pods
```

Apply:

```bash
oc apply -f hpa.yaml
```

---

## Step 3 — Verify HPA

```bash
oc get hpa -n i27-hpa-demo
```

Expected output:

```
NAME       REFERENCE             TARGETS   MINPODS   MAXPODS   REPLICAS   AGE
hpa-demo   Deployment/hpa-demo   8%/50%    1         5         1          30s
```

- `TARGETS` shows current CPU usage vs target (`8%/50%` means 8% actual, 50% threshold)
- `REPLICAS` shows current pod count

Describe the HPA for full details:

```bash
oc describe hpa hpa-demo -n i27-hpa-demo
```

Look for:

```
Metrics:    ( current / target )
  resource cpu on pods  (as a percentage of request):  8% (8m) / 50%
Min replicas:  1
Max replicas:  5
```

---

## Step 4 — Verify Metrics Server is Working

If TARGETS shows `<unknown>/50%`, the Metrics Server is not returning data yet. Wait 30–60 seconds after pod startup and check again:

```bash
oc adm top pods -n i27-hpa-demo
```

Once this returns CPU and memory values, the HPA TARGETS will populate correctly.
